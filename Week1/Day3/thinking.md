# 练习
## Part1
理解为什么要切片，以及不同切片策略的影响

### 理解
首先Embedding的输入有大小限制，其次长篇大论的内容插入，后续匹配时候命中的概率升高，但内容却不一定精准，譬如长篇大论讲RAG，最后提及一点prompt工程，后续用户提问prompt工程时候很有可能会把这个内容查出来，但核心还是讲RAG，意义不大。
切片策略有很多种，固定字数/固定行数/根据原文段落切分，不同场景使用不同的策略，固定字数和固定行数属于直接暴力切分，不考虑前后语境，容易出现一个词被拆入不同chunks了，通常需要加 overlap（重叠）带入少量前一个chunk的内容。根据原本段落切分，即尊重原文的分布拆解，这对原文的内容有一定的要求，若原文乱分段，东一句西一句，此策略也不适用。

### 举一反三
切片逻辑远不止这几种，这几种局限性太大了，知识库的内容来源也不仅仅是文章，还可能从网页，图片，视频，音频等等，对于这些多媒体的，这些就不适用。另外应该有能够根据简单语义拆分的策略/工具包等等。就光RAG向量数据库就可能存在多个垂直领域，譬如专门提供专业的RAG，提供源数据，提供切片能力，检索效果测评等等。

## Part2
把切片后的文档存入向量数据库，构建检索索引

### 理解
前期构建好RAG，后续可以动态维护，构建稳定的数据源头

**离线vs在线**：索引构建是离线做的（一次性&定期更新），查询时在线的
**增量更新vs全量更新**：知识库有新内容时，是追加新chunk还是全部重新切片+重新embedding？各有优劣
**Embiedding函数的一致性**：存入和查询必须用同一个Embedding模型，否则向量空间不匹配，索引完全失效

## Part3
串联检索和生成，跑通完整的RAG

### 理解
文档清洗 -> 文档切片 -> 插入RAG数据库（embedding） -> 用户提问 -> 检索Top-K -> 拼接Prompt -> 发送LLM模型 -> 响应用户

要手写接收用户问题，检索top-k，拼接Prompt以及发送LLM，同时还需要考虑top-k的k取值，太小可能会漏掉相关内容，太大会引入噪声增加token成本以及distcance阈值，若相差较大，则没有太大的参考意义


## Part4
直观对比有RAG和没RAG的回答质量差异

### 理解
有RAG的LLM回答内容可控，更准确，且对于些未公开的业务数据，LLM也能根据RAG的内容进行回答，只要prompt指令得当就能很大程度的避免幻觉问题。
无RAG的LLM虽然能通过prompt提高回答质量，但最终也可能存在幻觉问题，回答的质量也会参差不齐，取决于LLM的能力以及能使用的工具。


## Part5
体会不同的Prompt模板对RAG输出质量的影响

### 理解
不是说只要用了RAG知识库的LLM就一定不会出现幻觉，若仅仅是"请根据以下知识内容回答用户问题"还是很有可能出现幻觉的，要通过明确的指令，禁止LLM大模型胡编乱造，另外可以指定响应格式。

### 举一反三
根据不同的业务场景，设计不同的prompt，不是一定要严格根据RAG知识库回答，因为RAG知识库也有可能检索匹配度极低，但也提供给到LLM的情况（当然可以前置判断拦截），LLM也不一定不清楚用户问题的答案。
**Faithfulness（忠实度）vs Helpfulness（有用性）的权衡。严格限定只用检索内容回答（高 faithfulness），会牺牲一些有用性；允许 LLM 补充自身知识（高 helpfulness），则幻觉风险增加。实际项目中需要根据场景决定倾斜方向（医疗/法律等高风险场景倾向 faithfulness，闲聊/创意场景倾向 helpfulness）。**


# 核心要点理解和回答
1. 用户提问到回答，主要经过接收用户问题，检索Top-K，拼接Prompt，请求LLM，处理LLM响应，回复用户。当然还包括前期的RAG数据准备，包括数据清洗，切片，Embedding，落库。
2. 文本切片的原因其一是embedding的算法输入有限制，其二太大的文本容易导致是检索内容不精确，质量下降，。chunk_size太大容易导致检索质量下降，chunk_size太小容易出现知识内容不够充分，上下文不完整，没有足够的说服力。
3. 如果不加这种指令，就会出现LLM使用训练的知识进行回答，不准确，甚至幻觉
4. ~~有RAG的LLM回答延迟更低~~，RAG的延迟更高，因为要检索，输入成本相对较高，输出成本相对较低（如果无RAG的LLM没有特殊的Prompt的话），只要RAG知识库切片策略使用得当，知识点够细，回答质量要高于LLM，除此之外，只要及时喂最新的文档，就能回答LLM无法回答的内容（训练截止时间之后）。当然无RAG的LLM也不是一无是处，相对而言无RAG的LLM更通用，对于创意场景更适用。

**RAG vs 直接问 LLM 对比：**

| 维度 | RAG | 直接问 LLM |
|------|-----|-----------|
| 延迟 | 更高（多了检索步骤 + 更长的 input） | 更低 |
| 输入成本 | 更高（prompt 里塞了检索内容） | 更低 |
| 准确性 | 对知识库内的问题更准确、可溯源 | 可能幻觉 |
| 知识时效 | 可随时更新知识库 | 受限于训练截止日期 |
| 私有数据 | 可以接入企业数据 | 完全不了解 |
| 系统复杂度 | 更高（需维护向量库、Embedding 模型、切片策略） | 更低 |
| 适用场景 | 知识密集型问答、需要溯源、私有数据 | 通用问答、创意生成 |

5. ~~RAG和Fine-tuning微调，RAG是别人调好的模型，固定的，容易"雷同"，而Fine-tuning可以通过参数调整影响RAG的响应，当然不是一定会更优，也有可能更差，需要边调边观察。也就是说上来先用RAG试试效果如何，如果没有达到要求，再换成Fine-tunning进行调优。~~

纠正：RAG **不动模型**，是在推理时把外部知识检索出来塞进 Prompt，模型本身没有任何变化。Fine-tuning 是**修改模型权重**，让模型学习新的知识或行为模式。两者是独立的技术路线，也可以叠加使用。实际策略：先试 RAG，效果不够再考虑 Fine-tuning，或者两者结合。

**RAG vs Fine-tuning 对比：**

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 本质 | 不改模型，改输入（检索知识塞进 Prompt） | 改模型参数（用数据重新训练模型） |
| 知识更新 | 更新知识库即可，实时生效 | 需要重新训练，周期长 |
| 成本 | 运行成本（检索 + 更长的 prompt） | 训练成本（GPU 时间 + 数据标注） |
| 适用场景 | 知识密集型问答、需要频繁更新的场景 | 需要改变模型风格/格式/推理能力的场景 |
| 可溯源性 | 能指出答案来自哪段文档 | 不能溯源 |
| 实际策略 | 先试 RAG，不够再加 Fine-tuning，两者可叠加使用 | — |

# 思考不足
1. **搭建了 RAG 系统，但怎么知道它好不好？RAG 的评估有专门的框架和指标**：
- 检索质量：Recall@K（Top-K 里有没有包含正确答案的文档）、MRR（正确文档排在第几位）
- 生成质量：
  - Faithfulness（忠实度）：回答是否基于检索到的文档，有没有编造
  - Relevance（相关性）：回答是否切题
  - Completeness（完整性）：是否回答了问题的全部方面
- 评估工具：RAGAS 框架（专门用于 RAG 评估，自动化打分）
2. **没有思考上下文窗口与 RAG 的关系**：
现在的 LLM 上下文窗口越来越大（GPT-4o 128K、Claude 200K、Gemini 1M+），一个自然的问题是：如果窗口够大，还需要 RAG 吗？
答案是仍然需要，原因：
- 成本：把整个知识库塞进 prompt 太贵了（按 token 计费）
- Lost in the Middle：LLM 对超长上下文中间部分的关注度会下降，检索出精准片段比全塞进去效果好
- 延迟：input 越长，TTFT（首 token 延迟）越高
3. **没有讨论 Query 端的优化**
整篇笔记都在讨论文档端（切片、存储），但 RAG 的查询端也很重要：
- Query Rewriting（查询改写）：用户的原始问题可能表述模糊或口语化，用 LLM 改写成更适合检索的形式
- HyDE（Hypothetical Document Embeddings）：让 LLM 先生成一个"假想答案"，用这个假想答案做检索（而不是用原始问题），因为"答案"和"文档"的语义空间比"问题"和"文档"更接近
- Multi-Query：把一个问题拆成多个子问题分别检索，合并结果

# 面试高频考点

## 基础流程类
1. **请画出 RAG 的完整架构图，并解释每一步在做什么**
   - 索引侧：文档加载 → 清洗 → Chunking → Embedding → 存入向量库
   - 查询侧：用户提问 → Query Embedding → 向量检索 Top-K → （Rerank）→ Prompt 拼接 → LLM 生成 → 返回答案
2. **RAG 解决了 LLM 的哪些问题？**
   - 知识截止（Knowledge Cutoff）
   - 幻觉（Hallucination）
   - 私有数据不可达
3. **RAG vs Fine-tuning 的区别？什么时候用哪个？能叠加使用吗？**
   - RAG 改输入，Fine-tuning 改模型；可以叠加（用 Fine-tuning 训练模型更好地利用检索到的上下文）

## Chunking 类
4. **Chunking 策略有哪些？各有什么优劣？**
   - 固定大小、按段落/句子、递归切分（RecursiveCharacterTextSplitter）、语义切分（按 Embedding 相似度找分割点）
5. **chunk_size 和 overlap 怎么调？有没有最佳实践？**
   - 没有万能值，常见起步 chunk_size=500, overlap=50-100
   - 需要用评估指标（Recall@K）来验证不同配置的效果
6. **如果一个关键信息刚好被切在两个 chunk 的边界怎么办？**
   - Overlap 是第一道防线；Parent-Child Indexing（小 chunk 检索，返回大 chunk）是更高级的方案

## 检索与生成类
7. **Top-K 怎么定？检索到不相关的内容怎么处理？**
   - 设置 distance 阈值过滤；先大 K 检索再用 Rerank 精排缩小范围
8. **RAG 的 Prompt 模板设计有什么讲究？**
   - 角色设定、忠实度约束（不要编造）、格式要求、"不知道就说不知道"的兜底指令
   - system 和 user 角色分工
9. **RAG 场景下 temperature 应该设多少？为什么？**
   - 低 temperature（0 或 0.1），因为希望模型基于上下文回答，减少随机性
10. **如果 RAG 回答的质量不好，你会从哪些方面排查？**
    - Chunking 粒度 → Embedding 模型选型 → 检索参数（Top-K、阈值）→ 有没有 Rerank → Prompt 模板设计 → LLM 能力

## 进阶追问类
11. **什么是 Lost in the Middle？对 RAG 有什么影响？**
    - LLM 对长上下文中间部分的关注度低，检索结果的排列顺序会影响回答质量
    - 应对：把最相关的放在最前面或最后面，避免放在中间
12. **什么是 HyDE？它为什么能提升检索效果？**
    - 让 LLM 先生成假想答案，用答案做检索。因为"答案-文档"的语义距离比"问题-文档"更近
13. **RAG 系统怎么评估？你知道哪些评估指标？**
    - 检索侧：Recall@K、MRR、Precision@K
    - 生成侧：Faithfulness、Relevance、Completeness
    - 工具：RAGAS 框架
14. **上下文窗口越来越大（128K、1M），RAG 还有必要吗？**
    - 仍然需要：成本、Lost in the Middle、延迟三大原因
15. **Naive RAG、Advanced RAG、Modular RAG 的区别？**
    - Naive RAG：基础的检索+生成
    - Advanced RAG：加了 Pre-retrieval（Query 改写）、Post-retrieval（Rerank、压缩）
    - Modular RAG：可插拔的模块化架构，灵活组合检索、路由、生成等组件
