# 练习

## Part1
理解 BM25（传统关键词检索）的工作原理

### 理解
BM25 是经典的信息检索算法，基于词频（TF）和逆文档频率（IDF）的改进版，加了长度归一化和 TF 饱和函数。中文场景下需要先用 jieba 分词，分词质量直接影响检索效果。

BM25 做的是**词级精确匹配**：搜"鸡蛋"就只匹配包含"鸡蛋"的文档，搜"蛋"不一定能匹配"鸡蛋"。优点是对专业术语、编号等精确关键词检索效果好，缺点是无法理解同义词（"存放"搜不到"保存"）。

### 关键区分
向量检索的弱点不是"找不到"，而是**容易返回语义相似但不够精确的结果**。比如搜"鸡蛋保存"可能同时返回"鸭蛋保存"、"鸡蛋变质判断"等语义相近但不够精确的内容。在需要字面精确匹配的场景下（编号、型号、否定句），BM25 的区分度更高。


## Part2
把向量检索和 BM25 结合起来，取长补短

### 理解
混合检索的核心思路：分别用向量检索和 BM25 检索各拿 Top-K 个结果，然后用 RRF（Reciprocal Rank Fusion）合并排名。

RRF 公式：`score(d) = Σ 1/(k + rank_i(d))`，其中 k 通常取 60。

**为什么用 RRF 而不是直接加分数？** BM25 的分数和向量检索的距离量纲完全不同（一个是分数越高越好，一个是距离越小越好），直接相加没有意义。RRF 只用排名，绕过了量纲问题。

### 关于索引构建
混合检索意味着从索引构建阶段就是两条线并行维护的：
```
原始文档 → 切片
  ├── Embedding → 存入向量库（向量索引）
  └── 分词     → 构建 BM25（关键词索引）
```
同一批 chunk，同时建两套索引。查询时两套索引各搜一遍，RRF 合并结果。

### BM25 的索引生命周期
Day4 代码中 BM25 索引是纯内存的（`BM25Okapi` 对象），进程退出就丢失。生产环境通常用 Elasticsearch 代替，自带 BM25 实现、数据持久化、支持增量更新和分布式。


## Part3
理解 Rerank 的原理和作用

### 理解
Rerank 的核心思想是「召回-排序」两阶段架构：
1. 初步检索（向量/BM25/混合）拿到一批候选文档（如 Top-20）
2. 用 Cross-Encoder 逐一判断「查询-文档」的相关性
3. 根据精细分数重新排序，取 Top-K

### Bi-Encoder vs Cross-Encoder
- **Bi-Encoder**（向量检索）：query 和 doc 分别编码成向量，再算余弦相似度。**快但粗**。
- **Cross-Encoder**（Rerank）：query 和 doc 拼接在一起输入模型，输出相关性分数。**慢但准**。

### 为什么不直接用 Cross-Encoder 做全量检索？
Cross-Encoder 需要把 query 和每个文档逐对输入模型打分。10 万篇文档就要跑 10 万次模型，太慢了。就像查字典：先用拼音找到对应那片区域（粗排/召回），然后再逐页翻找正确的字（精排/Rerank）。直接从头翻到尾，耗时太长。


## Part4
把混合检索 + Rerank + LLM 生成串在一起

### 理解
完整的优化版 RAG 流水线：

```
用户提问
  ↓
Stage 1: 召回（混合检索）  → 目标：高 Recall（宁可多不可漏）
  ├── 向量检索 Top-10（语义匹配）
  ├── BM25 Top-10（关键词匹配）
  └── RRF 融合排序 → 候选集 Top-10
  ↓
Stage 2: 精排（Rerank）    → 目标：高 Precision（把最相关的排到最前面）
  └── Cross-Encoder 逐对打分 → Top-3
  ↓
Stage 3: 生成（LLM）       → 目标：高 Faithfulness（忠实于检索内容，不编造）
  └── Top-3 文档 + 用户问题 → Prompt → LLM → 回答
```

每个阶段有明确目标，这就是工程化的 RAG 系统。
别忘了还有离线侧：文档加载 → 清洗 → 切片 → 同时构建向量索引和 BM25 索引。


## Part5
学会用简单指标评估 RAG 检索质量

### 理解
- **Hit Rate@K**：Top-K 中至少命中一个正确文档的比例 → 衡量"有没有找到"
- **MRR@K**（Mean Reciprocal Rank）：正确文档排在第几位的倒数的平均值 → 衡量"排得准不准"
  - 正确文档排第1位，贡献 1/1 = 1
  - 正确文档排第3位，贡献 1/3 ≈ 0.33

### 评估的前提
**评测集是地基**。需要人工标注：每个测试问题应该命中哪些 chunk。这是一次性投入，后续所有实验都复用。没有评测集，所有调优都是凭感觉。


# 核心要点回答

1. **BM25 vs 向量检索**：BM25 做词级精确匹配（快、精确、不需要 GPU），但不理解同义词；向量检索做语义匹配（能找到同义词），但可能返回语义相似却不够精确的结果。混合使用是为了互补——向量保证语义覆盖不漏，BM25 保证关键词精确不乱。

2. **RRF 的思路**：只用排名来融合，不用原始分数。因为 BM25 分数和向量距离的量纲不同，直接相加没有意义。RRF 公式 `1/(k + rank)` 把不同方法的排名统一到同一个可比的尺度上。

3. **Cross-Encoder vs Bi-Encoder**：Bi-Encoder 分别编码，快但粗；Cross-Encoder 拼接编码，慢但准。不能直接用 Cross-Encoder 做全量检索，因为 10 万文档要跑 10 万次模型，速度不可接受。

4. **Hit Rate** 衡量"有没有找到"（Top-K 命中率），**MRR** 衡量"找到的排在多前面"（排名质量）。

5. 完整流程见 Part4 的流程图。三阶段目标：召回追求高 Recall（不漏），精排追求高 Precision（排准），生成追求高 Faithfulness（不编）。


# 调优方法论

## 核心原则：分层评估，控制变量

前期需要人工标注的方式明确检验标准（评测集），再通过控制变量法进行分层调优。

### 每一层的独立评估方法

| 层级 | 评估方式 | 不依赖下游 |
|------|---------|-----------|
| 切片 | 人工抽检语义完整性 + 固定后续环节只换切片策略对比 Hit Rate | ✅ |
| 检索 | Hit Rate@K / MRR@K | ✅ |
| Rerank | 对比精排前后的 MRR 变化 | ✅ |
| 生成 | 固定检索结果，只评估 LLM 输出（Faithfulness / Relevance） | ✅ |

### 调优顺序：从前往后

```
Step 1: 构建评测集（一次性基础设施）
Step 2: 只调切片 → 用 Hit Rate 判断哪种策略好
Step 3: 固定切片，调检索策略 → 对比 向量/BM25/混合 的指标
Step 4: 固定检索，加 Rerank → 看 MRR 有没有提升
Step 5: 固定检索+Rerank，调 Prompt → 看生成质量
Step 6: 跑端到端评估，看最终效果
```

**每次只动一个变量，其他全部固定**（控制变量法）。前面的环节决定了后面的上限——切片不好，后面再怎么调都救不回来。

### 排查 bad case：逐层检查输出

1. 检索 Top-K 里有没有正确文档？ → 没有 → 切片/Embedding 模型/检索策略的问题
2. 有但排名靠后？ → Rerank 的问题
3. 有且排在前面，但回答还是不对？ → Prompt / LLM 的问题
