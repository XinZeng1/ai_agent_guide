## Part1-5
手写ReAct的核心是"thought -> action -> observation -> thought -> action ... -> final answer"。详细的流程是：LLM -> 思考 -> 判断是否能回答用户问题 -> 不能则判断是否调用工具 -> 调用工具 -> 把工具作为输入提供给下一轮LLM -> 思考... 重复上述的流程。
ReAct从prompt设计上就需要这么去设计，
prompt要注意的点：
1. 要明确输出格式（方便后续解析），需要有思考，动作（工具），观察，另外要明确能够回答用户问题时就输出结果
2. 要明确严格按照工具说明与实际问题有关再调用工具，避免乱调用
代码工程上：
1. 工具说明要精准，入参要明确，出参要尽可能统一
2. 要设计最大循环次数，避免过多次循环
3. 可以先判断是否有结果，没有再调用工具
疑问点：
1. 思考那个动作是否可以忽略，首先那个用户不敢知，其次只要动作和观察的点以及终止条件即可。 A: 便于观察每一步的动作，可以快速定位问题
2. 观察那个点一定是"Observation"字符串？或者说为什么"Observation"这个能作为截停LLM输出的关键标识，我看prompt没有设计这个单词的时候，LLM模型调用的时候也可以传这个作为截停的标识。

A:
1. ReAct是一种具备一定思考能力的简单Agent设计范式或流程，ReAct主要解决LLM只有头脑，没有手脚的问题，给LLM拼手脚，然后通过手脚去做事。核心循环是 思考->动作->观察->思考->动作....->直到能够回答用户问题
2. CoT仅具备思考，即根据训练数据进行回答用户问题，ReAct可以根据用户问题使用现有的工具补充问题的相关内容后综合回答用户，可以使用训练之外的数据，实时性很高，也可以通过工具多媒体响应
3. 理论上什么类型都可以，只要处理工具消息足够适配（将工具的输出转换为LLM支持的格式，文字）就可以。当然为了统一和简单，通常都是字符串，或json schema格式
4. 通过工程上DAG设计实现并发调用
5. 整体链路 思考 -> 动作 -> 观察。要设计最大循环次数，设置终止条件，工具考虑适配器模式，做成可插拔式，工具的输出要一致，工具调用可以考虑用DAG设计，提高并发和解决工具依赖的问题，当然Prompt要对工具部分做好限制，避免工具做一些危险动作。

疑问1：Thought能不能去掉？
你的分析"便于观察定位问题"是对的，但还有一个更深的原因：Thought实际上会提升LLM的推理质量。这就是CoT（Chain of Thought）的核心发现——让模型"说出"推理过程，会让最终决策更准确。去掉Thought，LLM直接输出Action，它的工具选择准确率会下降。你可以自己做个实验：同一个问题，一个Prompt要求先Thought再Action，另一个直接要求输出Action，对比调用准确率。
疑问2：为什么"Observation"能作为stop标识？
这是一个很好的技术细节问题。stop=["Observation:"] 的作用是告诉API：当LLM生成到这个字符串时立即停下来，不要继续生成。这不是因为"Observation"有什么魔力，而是一个工程设计选择——你用任何字符串都行，比如 stop=["[TOOL_RESULT]"] 也完全可以。
用"Observation"是因为ReAct的Prompt模板里定义了这个格式约定。LLM在生成时会按照Prompt模板的格式继续写，它在写完Action Input之后自然会想写"Observation:"然后编造一个结果。stop参数在这里截停它，防止它自己编造工具结果。然后你的代码执行真正的工具，把真实结果作为"Observation: xxx"拼接回去。本质上这是在抢夺控制权——LLM想自己编结果，你用stop参数拦住它，用真实结果替代。

思考题点评
第1题，"给LLM拼手脚"这个比喻很生动，面试时可以用。但"简单Agent设计范式"这个定位有点低估了——ReAct是Agent领域最被广泛引用的论文之一，它不是"简单"的范式，而是"基础"的范式。很多复杂Agent框架的底层都是ReAct循环。
第2题，说得清楚。补一个面试加分的细节：CoT在某些纯推理任务上效果反而比ReAct好（比如数学证明），因为不需要外部信息，加工具反而是干扰。所以不是"ReAct一定比CoT好"，而是需要外部信息时ReAct更好，纯推理时CoT可能更好。
第3题，你说"理论上什么类型都可以"——这个回答如果放在面试里会有点模糊。面试官问这个问题期待的精确答案是：LLM返回的是JSON格式的结构化数据（函数名+参数），不是工具的执行结果。 你的回答偏向了"工具输出是什么类型"，但题目问的是"LLM调用工具后返回的是什么类型"，指的是LLM那一侧的输出格式。这两个方向都对，面试时建议先答LLM侧（JSON），再补充工具侧（通常统一为字符串/JSON）。
第4题和第5题，都很好。第5题你提到的"适配器模式、可插拔式"特别好，这是后端工程师的优势体现。面试时主动提这些设计模式会让面试官觉得你有工程落地能力，不只是跑了个Demo。
